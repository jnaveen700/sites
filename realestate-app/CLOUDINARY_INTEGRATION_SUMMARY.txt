🔄 COMPLETE UPLOAD FLOW (End-to-End):

ADMIN UPLOAD FLOW:
1. Admin selects saree images on phone (3 photos, ~8MB each)
2. Frontend AddSaree.jsx validates: file type, size (<5MB)
3. Frontend shows preview thumbnails
4. Admin enters saree details: name, price, material, etc.
5. Admin clicks "Add Saree" button

REQUEST CHAIN:
6. Browser sends POST /api/sarees with multipart/form-data
7. authMiddleware verifies JWT token
8. authorizeRoles('admin') checks if user is admin
9. uploadMiddleware.array('images', 10) intercepts request:
   - Validates each file (type: JPEG/PNG/WebP, size: <5MB)
   - Streams to Cloudinary
   - Cloudinary applies transformations:
     * Width: 1200px (resize if wider)
     * Quality: auto (80% JPEG quality)
     * Format: auto (WebP if modern browser, else JPEG)
   - Result: 8MB image → 1.2MB optimized
   - Returns: req.files with Cloudinary URLs + public_ids

CONTROLLER PROCESSING:
10. sareeController.createSaree() receives request
11. Validates form fields (prices, stock, etc.)
12. Maps req.files to images array:
    { url: Cloudinary URL, public_id: for deletion }
13. Creates MongoDB document with images array
14. Saves to database

ERROR HANDLING:
15. If MongoDB save fails:
    - Catch block loops through req.files
    - Calls deleteImageFromCloudinary(public_id) for each
    - Removes orphaned images from Cloudinary
    - Returns 500 error to frontend
    - Admin can safely retry without duplicates

RESPONSE:
16. Returns 201 with created saree object
17. Frontend shows "Saree added successfully!"
18. Redirects to manage sarees page

📁 MONGODB SCHEMA STRUCTURE:

After upload, saree document looks like:
{
  _id: ObjectId(...),
  designName: "Red Silk Saree",
  retailPrice: 2500,
  wholesalePrice: 1800,
  stock: 50,
  images: [
    {
      url: "https://res.cloudinary.com/xyz/image/upload/w_1200,q_auto,f_auto/saree-products/abc123.jpg",
      public_id: "saree-products/abc123"
    },
    {
      url: "https://res.cloudinary.com/xyz/image/upload/w_1200,q_auto,f_auto/saree-products/def456.jpg",
      public_id: "saree-products/def456"
    }
  ],
  createdBy: ObjectId(admin_id),
  createdAt: ISODate("2024-05-23T10:30:00Z")
}

KEY POINTS:
- url: served to customers (already optimized, 1.2MB)
- public_id: stored for deletion when editing/removing images
- Max 10 images per saree (validated by Cloudinary schema)

✨ IMAGE DELETION FLOW:

When admin deletes a saree or removes an image:
1. Admin clicks delete on saree or image
2. Controller receives request with saree ID
3. Fetches saree from MongoDB
4. Loops through images array
5. For each image: calls deleteImageFromCloudinary(public_id)
6. Cloudinary finds image by public_id in saree-products folder
7. Removes image from Cloudinary storage
8. Removes from images array in MongoDB
9. Saves updated saree
10. Returns success response

Result: No orphaned images, no wasted storage
